# Word To Product Superapp Command

## Objetivo

Definir de forma explicita como crear una super app capaz de transformar pocas palabras en:

- apps
- juegos
- paginas web
- automatizaciones
- agentes
- contenido
- video
- paneles
- sistemas internos

## Orden canónica

La super app no se construye como un generador suelto.
Se construye como un sistema operativo de creacion.

## Promesa del producto

El usuario escribe una idea corta.
La plataforma:

1. entiende la intencion
2. completa el contexto faltante
3. genera spec
4. arma arquitectura
5. crea estructura base
6. produce modulos
7. valida
8. deja listo para iterar, publicar o delegar

## Lo que debe tener si o si

### 1. Intent Engine

Convierte palabras cortas en intencion estructurada.

Debe extraer:

- tipo de producto
- usuario objetivo
- objetivo principal
- plataformas
- nivel de complejidad
- estilo deseado
- restricciones

### 2. Spec Engine

Genera automaticamente:

- product brief
- feature map
- flows
- data model
- architecture draft
- release path
- risk map

### 3. Blueprint Engine

Crea estructuras listas para usar:

- frontend shell
- backend contracts
- database schema inicial
- modulos
- design system base
- templates

### 4. Assembly Engine

Construye y conecta:

- rutas
- componentes
- servicios
- adapters
- auth
- permisos
- observabilidad
- tests minimos

### 5. Agent Orchestrator

Reparte el trabajo entre agentes por:

- arquitectura
- frontend
- backend
- datos
- growth
- QA
- media

### 6. Validation Engine

Todo lo que se crea debe pasar por:

- typecheck
- build
- contratos
- fallback
- seguridad minima
- smoke tests

### 7. Runtime Memory

Debe guardar:

- decisiones
- templates usados
- aprendizajes
- errores previos
- patrones exitosos

### 8. Delivery Engine

Debe poder dejar salida en:

- repositorio listo
- modulo listo
- branch lista
- handoff claro
- release candidate

## Modos del sistema

### Modo 1: Idea to spec

Para convertir una idea corta en plan profesional.

### Modo 2: Spec to scaffold

Para generar estructura inicial completa.

### Modo 3: Module builder

Para crear o reemplazar modulos puntuales.

### Modo 4: Repair and optimize

Para arreglar, limpiar y endurecer.

### Modo 5: Launch and scale

Para preparar crecimiento, marketing, analytics y operaciones.

## Tipos de salida que debe soportar

- landing page
- web app
- mobile app
- game prototype
- admin system
- AI room
- workflow automation
- content system
- video pipeline

## Guardrails obligatorios

1. no crear sin spec
2. no tocar codigo critico sin task
3. no dejar modulos huerfanos
4. no soltar codigo sin validacion
5. no exponer secretos
6. no confundir inferencia con hecho
7. no cerrar una tarea sin salida formal

## Preguntas obligatorias antes de construirla

1. que significa "crear todo" dentro del alcance real
2. que salidas tienen prioridad
3. que partes son generativas y cuales son deterministicas
4. que validaciones son bloqueantes
5. como se versiona lo generado
6. como se evita codigo basura
